이번 작업의 목표는 두 가지였다. 첫째는 OL HOME 안에 `NET` 메뉴를 실제 탐색 화면으로 세우는 것, 둘째는 여기서 만든 이름카드 경험을 `TEXT`와 `STORY` 읽기 화면으로 옮기는 것이었다.

결론부터 말하면, 이번 1차 범위에서는 NET 메인페이지와 탐색페이지의 기본 구조를 완료했고, 이름카드 목록과 상세 패널의 동작 원칙도 정리했다. 이어서 같은 구조를 TEXT와 STORY 상세 페이지 우측 사이드에 붙여, 문서를 읽다가 바로 연결된 인물·장소·개념 이름카드를 열어볼 수 있게 만들었다.

## NET을 별도 메뉴로 분리한 이유

기존 `ENTITY` 상세 페이지는 개별 문서를 읽는 구조였다. 하지만 이름을 모아 보고, 유형으로 좁히고, 관련 대상을 비교하는 흐름은 별도의 탐색 화면이 필요했다.

그래서 NET은 기존 Entity 시스템을 대체하는 것이 아니라, 그 위에 얹는 별도 탐색 레이어로 잡았다.

- `/entity/...`는 개별 이름카드의 문서형 상세
- `/net/`은 NET의 진입 메인페이지
- `/net/explore/`는 이름카드 탐색 전용 화면

이 분리는 중요했다. 그래야 문서 읽기와 탐색 경험을 섞지 않고 각각의 밀도를 유지할 수 있다.

## `/net`과 `/net/explore`

`/net`은 진입 페이지다. 검색 입력과 빠른 유형 진입을 중심에 두고, NET이 어떤 메뉴인지 짧게 설명하는 역할만 맡겼다.

실제 탐색 기능은 `/net/explore`에 모았다. 여기서는 3열 구조를 사용한다.

- 좌측: 유형 및 prefix 태그 필터
- 중앙: 이름카드 목록
- 우측: 선택한 이름카드 상세 패널

이 구조는 ENTITY 문서를 그대로 늘어놓는 방식보다 훨씬 분명했다. 사용자는 중앙에서 이름을 훑고, 우측에서 선택 대상을 바로 확인한다. 문서 전체 페이지로 이동하지 않아도 기본 맥락을 파악할 수 있다.

이번 1차 구현에서 NET은 여기까지를 완료 범위로 봤다. 관계망 그래프와 지도 탭은 의도적으로 뒤로 미뤘다. 먼저 카드 탐색 경험이 성립하는지 확인하는 편이 맞았기 때문이다.

## 이름카드 기능을 어떻게 구체화했는가

이번 작업의 핵심은 이름카드를 단순 링크가 아니라, **목록과 상세를 오가는 작동 단위**로 만든 것이다.

NET 탐색화면에서는 중앙 목록에서 카드를 선택하면 우측 상세 패널이 바뀐다. 카드에는 이름, 유형, 기본 설명, 태그가 보이고, 상세 패널에서는 더 긴 설명과 관련 콘텐츠로 이동할 수 있다.

여기서 중요한 판단은 “목록과 상세를 동시에 과하게 펼치지 않는다”는 점이었다. NET에서는 중앙과 우측이 분리되어 있으니 목록과 상세를 나눌 수 있지만, 문서 읽기 화면에서는 그렇게 하면 폭이 금방 무너진다.

이 판단이 다음 단계, TEXT/STORY 적용으로 그대로 이어졌다.

## TEXT와 STORY에서 이름카드를 호출하는 방식

이번에 TEXT와 STORY의 문서-level Entity 연결 방식도 정리했다. 과거에는 TEXT와 STORY가 서로 다른 frontmatter 이름을 쓰고 있었는데, 이제 둘 다 아래처럼 통일했다.

```yaml
entities:
  - "five-aggregates"
  - "twelve-sense-bases"
```

이 배열에는 인물, 장소, 개념을 구분 없이 넣는다. 실제 타입은 각 Entity 문서의 `type`에서 해결한다. 문서 안에서 보여주는 순서도 이 배열 순서를 그대로 따른다.

이 통일 덕분에 TEXT와 STORY가 같은 우측 이름카드 패널 컴포넌트를 재사용할 수 있게 됐다.

## TEXT 상세 페이지: 2단에서 3단으로

TEXT 상세 페이지는 기존의 2단 구조에서 3단 구조로 바꿨다.

- 좌측: 시리즈/문서 목록
- 중앙: 본문
- 우측: 해당 문서에 연결된 이름카드 패널

우측 패널에는 문서 frontmatter의 `entities`에 적힌 이름카드만 나타난다. 자동 추론은 하지 않는다. 이 원칙을 지킨 이유는, 문서 저자가 “이 문서와 정말 연결된 이름”만 선택해서 보여주게 하기 위해서다.

테스트용으로 `불교의 세계관 - 두 가지 관점에 근거한 나와 세계에 대한 이해` 문서에 `오온(five-aggregates)`과 `십이처(twelve-sense-bases)` 이름카드를 새로 만들어 연결했다.

이제 본문을 읽다가 우측에서 두 개념 이름카드를 바로 열 수 있다.

## STORY 상세 페이지도 같은 패턴으로 확장

STORY에도 같은 우측 패널을 붙였다. 다만 STORY는 PART 네비게이션과 본문 흐름이 이미 있어서, 모바일에서는 3단 구조가 깨지지 않도록 반응형 우선순위를 다시 잡아야 했다.

이 과정에서 데스크탑용 그리드 규칙의 specificity가 모바일 규칙을 덮어쓰는 문제가 있었고, 별도 모바일 override를 추가해 다시 단일 컬럼으로 접히도록 정리했다.

즉, 이번 STORY 적용은 단순 이식이 아니라 “읽기 UI를 망치지 않는 범위 안에서 공용 패널을 넣는 작업”에 가까웠다.

## 문서형 Entity 상세도 함께 정리했다

이름카드 시스템을 확장하면서 기존 Entity 상세 페이지의 문제도 드러났다. 특히 하단 참조 링크와 역참조 링크가 raw id 기반으로 출력되고, 실제 링크 경로도 잘못되어 404가 나는 문제가 있었다.

그래서 relation 대상 Entity를 다시 resolve해서:

- 링크는 실제 route인 `/entity/{type}s/{id}/`로 연결하고
- 표시 텍스트는 `dependent-origination` 같은 id가 아니라 `연기`, `나가르주나` 같은 이름카드 제목으로 바꿨다

모바일 Entity 상세도 같이 다듬었다. 브레드크럼은 최상단으로 올리고, 중복되는 제목/메타는 숨겨서 작은 화면에서 같은 정보를 두 번 보지 않게 했다.

## 지금 멈춘 지점

현재 판단으로는 NET 메인페이지와 탐색페이지는 1차 완료 상태다. 이름카드 목록과 상세 패널의 구조, 문서 연결 방식, TEXT/STORY 호출 방식까지 기본 축은 잡혔다.

아직 남은 단계는 분명하다.

- Concept Entity 스키마 추가 확장
- NET 관계망 그래프 1차 구현
- 지도 탭 placeholder와 수작업 SVG 지도 연결

다만 이 기능들은 지금 바로 얹기보다, 먼저 “문서 안에서 이름을 읽고, 탐색하고, 다시 문서로 돌아오는 흐름”이 충분히 안정적인지 본 뒤 진행하는 편이 맞다.

이번 1차 구현은 그 기반을 만드는 작업이었다.
