오늘 작업의 중심은 OL HOME의 TEXT와 STORY 페이지를 단순 목록에서 "시리즈 단위로 읽고 탐색하는 콘텐츠 리더"로 바꾸는 것이었다. 동시에 DESIGN 콘텐츠도 `src/content`를 원본으로 삼고 빌드 시 `public/generated`로 복사하는 흐름 안에 새 항목을 추가했다.

## TEXT — 메인 페이지와 시리즈 페이지 분리

TEXT는 "경전·논서·선어록·강의록을 번역하고 주석한 문헌 콘텐츠"로 정리했다. 기존에는 목록과 문서 읽기 경험이 한 페이지 성격 안에 섞여 있었지만, 이제는 메인 페이지와 시리즈 페이지의 역할을 나누었다.

메인 페이지는 시리즈 카드 리스트를 보여준다. 각 시리즈는 `src/content/text/<series>/index.md`를 메타데이터 원본으로 삼고, 실제 문서는 시리즈 폴더 안의 `part/` 아래에 둔다.

```
src/content/text/
  body-mind-transformation/
    index.md
    assets/
    part/
      01-six-steps.md
```

이 구조 덕분에 시리즈 카드에 표시되는 제목, 설명, 태그, 상태, 대표 이미지를 문서 목록과 분리해서 관리할 수 있다. 문서 자체는 계속 part, group, chapter 위계에 따라 정리된다.

## STORY — 책이 아니라 시리즈 리더로 보기

STORY는 "불교의 인물·설화·가르침을 이야기 형식으로 재구성한 서사 콘텐츠"로 정리했다. 구조는 TEXT와 맞추되, 읽기 방식은 STORY의 성격에 맞게 다르게 설계했다.

STORY 시리즈 페이지는 상단에 시리즈 제목을 중앙 배치하고, 하단에 컨테이너형 리더를 둔다. 첫 화면은 서지정보 탭이다. 이 탭은 커버, 문서목록, 서지정보의 3단 카드 레이아웃으로 바꾸었다. 모바일에서는 문서목록 카드와 서지정보 카드를 접고 펼 수 있게 했다.

PART 본문 탭은 왼쪽 사이드바와 오른쪽 본문으로 나뉜다. 사이드바에는 해당 PART 안의 그룹과 문서만 출력한다. 본문에는 한 번에 하나의 문서만 표시되도록 수정했다. 이전에는 한 PART의 모든 문서가 이어져 출력되어 긴 문서 리더로는 부담이 있었다.

## 태그 체계 — prefix 기반으로 통일

TEXT와 STORY 모두 태그를 단순 문자열 목록이 아니라 prefix 기반 태그로 정리하기 시작했다.

```yaml
tags:
  - "인물/아나타삔디까"
  - "장소/라자가하-왕사성"
  - "개념/연기"
tagAliases:
  "인물/아나타삔디까": ["Anāthapiṇḍika", "급고독장자"]
```

이 방식은 태그가 많아질수록 중요해진다. 하나의 긴 태그 드롭다운에 모든 태그를 넣는 대신, 현재 문서에 실제로 존재하는 prefix만 자동으로 분류해 필터 버튼을 만든다. 예를 들어 인물 태그가 있으면 `인물` 필터가 생기고, 문헌 태그가 없으면 `문헌` 필터는 만들지 않는다.

이후 다중 필터도 적용했다. 선택한 태그들을 AND 조건으로 조합해, 여러 기준을 동시에 만족하는 문서만 남긴다. 태그가 500개 이상의 STORY 문서에 붙기 시작하면 단일 태그 검색보다 이런 조합 검색이 훨씬 중요해진다.

## 태그 인덱스와 검색 패널

STORY 시리즈 페이지에는 prefix별 태그 인덱스와 검색 결과 패널을 추가했다. 태그를 선택하면 해당 태그와 관련된 문서 목록이 part, group, chapter 구조를 유지한 채 표시된다.

TEXT 시리즈 페이지에도 같은 방식의 태그 필터와 검색 패널을 확장했다. 두 리더가 완전히 같은 UI일 필요는 없지만, 문서를 찾는 방식은 통일되어야 한다. 사용자는 TEXT든 STORY든 "시리즈 안에서 태그로 문서를 좁히고, 결과에서 바로 문서로 이동한다"는 같은 경험을 갖게 된다.

## 모바일 도구 — 목록, 태그, 목차, 복사

STORY 모바일에서는 데스크탑 사이드바를 그대로 위에 쌓는 방식이 적합하지 않았다. 그래서 모바일 전용 도구 줄을 만들었다.

```
[목록] [태그] [On this page] [Copy]
```

목록은 현재 PART의 문서 목록을 열고, 태그는 prefix별 태그 필터를 연다. On this page는 현재 문서의 본문 목차를 표시하고, Copy는 링크 복사와 마크다운 본문 복사를 제공한다.

TEXT 세부 문서 페이지도 Copy 도구를 같은 방향으로 맞추었다. "마크다운 복사"는 단순한 마크다운 링크 복사가 아니라, 본문 내용을 마크다운으로 복사하는 기능으로 바꾸었다.

## 이전 문서, 다음 문서

TEXT와 STORY 본문 하단에는 이전 문서와 다음 문서 버튼을 추가했다.

TEXT는 실제 상세 문서 페이지가 분리되어 있으므로 링크로 이동한다. STORY는 하나의 시리즈 페이지 안에서 문서가 전환되는 구조이므로 버튼을 클릭하면 해당 PART 탭으로 이동하고, 그 안의 문서를 활성화한다.

이전·다음 문서는 독서 흐름을 유지하는 최소 장치다. 특히 STORY처럼 문서 수가 많아질 콘텐츠에서는 사이드바만으로 이동하는 것보다 하단 탐색이 훨씬 자연스럽다.

## 콘텐츠 자산 관리 — src/content를 원본으로

TEXT, STORY, DESIGN 모두 실제 콘텐츠 자산은 `src/content` 안에 두고, 빌드 시 `public/generated`로 복사하는 구조를 유지했다.

오늘 DESIGN에는 `two-perspectives` 항목을 추가했다.

```
src/content/design/two-perspectives/
  index.md
  assets/
    index.html
    preview1.png
    thumb.jpg
```

빌드 후에는 다음 경로에 산출물이 생성된다.

```
public/generated/design/two-perspectives/
```

이 구조는 원본과 배포 산출물을 분리한다. 사람이 관리하는 파일은 `src/content`에 있고, 웹에서 직접 접근하는 파일은 빌드 결과로 생성된다.

## 보류한 실험 — STORY reader sticky

STORY 시리즈 페이지에서 `.ol-story-reader-shell`과 사이드바를 sticky로 고정하는 실험도 진행했다. CSS만으로 몇 가지 방식을 테스트했지만, 실제 스크롤 감각이 매끄럽지 않았다.

특히 shell 전체를 고정하려면 내부 패널을 별도 스크롤 컨테이너로 만들어야 했고, 이 방식은 STORY 리더의 자연스러운 읽기 흐름과 충돌했다. 결국 sticky 적용은 보류하고 기존 구조로 되돌렸다.

이번 결론은 명확하다. 긴 서사 문서에서는 "고정된 도구"보다 "방해 없는 읽기 흐름"이 우선이다. 필요하면 나중에 CSS sticky가 아니라, 별도 floating control 또는 JS 기반의 가벼운 보조 네비게이션으로 다시 검토하는 편이 낫다.

## 다음 과제

오늘 작업으로 TEXT와 STORY는 공통된 콘텐츠 운영 원칙을 갖게 되었다.

- 시리즈 메타데이터는 각 시리즈의 `index.md`에서 관리한다.
- 문서는 part, group, chapter 위계로 정리한다.
- 태그는 prefix 기반으로 분류한다.
- 시리즈 페이지 안에서 태그 검색과 문서 탐색을 제공한다.
- 모바일은 데스크탑 사이드바를 축소하지 않고 별도 도구로 재구성한다.

다음 단계는 실제 콘텐츠가 많아졌을 때의 압력 테스트다. 특히 STORY는 500개 이상의 문서를 전제로 하고 있으므로, 태그 인덱스의 정렬, 검색 결과 표시 방식, 문서 전환 속도, 모바일 패널 사용성을 계속 점검해야 한다.
