# 프로젝트 코드 리팩터링 로드맵 v0.1

문서 버전: v0.1  
작성일: 2026-06-25  
업데이트일: 2026-06-25  
대상 프로젝트: OL HOME (`biwoom.github.io`)  
분석 범위: 스타일, 스키마, 콘텐츠 작성 규칙, 콘텐츠 내부 이미지 자산 연결 방식, frontmatter 일관성, 저장소 위생, 검색/필터 UI, 운영 문서 정합성  

---

## 0. 문서 목적

이 문서는 OL HOME의 현재 코드 상태를 바탕으로, 유지보수성과 일관성을 높이기 위한 리팩터링 우선순위와 실행 계획을 정리한다.

이번 로드맵은 다음 기준을 전제로 한다.

```txt
1. 지금 당장 사이트가 깨져 있지는 않다.
2. 그러나 운영 규칙과 실제 코드가 분기되기 시작했다.
3. 이 상태를 방치하면 새 콘텐츠가 늘수록 스키마 혼선과 예외 처리 비용이 커진다.
4. 목표는 "문제가 생기면 고치는 프로젝트"가 아니라
   "새 작업을 추가해도 구조적 품질이 무너지지 않는 프로젝트"다.
```

핵심 원칙은 다음과 같다.

```txt
코드와 문서의 정합성
컬렉션별 규칙의 통일
자동 검증 가능한 운영 구조
중복 로직 축소
콘텐츠 작성자 관점의 단순성
```

---

## 1. 현재 판단 요약

현재 프로젝트는 다음 점에서 기반이 좋다.

```txt
Astro content collection 기반 구조가 명확하다.
src/content.config.ts가 메타데이터 스키마의 단일 진실 원천 역할을 한다.
DESIGN/STORY 자산을 public/generated로 증분 동기화하는 운영 모델이 있다.
TEXT/STORY/DESIGN/BLOG/ENTITY의 역할 구분이 비교적 선명하다.
Pagefind 메타와 검색 인덱스 분리 방향이 잡혀 있다.
```

반면, 다음 문제는 장기 운영 리스크로 판단한다.

```txt
1. 코드 규칙과 운영 문서가 서로 다르다.
2. 이미지 자산 연결 규칙이 컬렉션마다 다르다.
3. 빌드 외의 검증 체계가 부족하다.
4. TEXT/STORY/TAG 유틸이 중복 구현되어 있다.
5. 필터 UI와 클라이언트 스크립트가 분산되어 있다.
6. 저장소 위생 관리가 느슨하다.
7. 스타일 시스템 대비 인라인 스타일 비중이 높다.
8. 외부 폰트/리소스 의존이 다소 무겁다.
```

이 8개 항목을 기준으로 아래 로드맵을 제안한다.

## 3. 항목별 리팩터링 로드맵

### 3.1 운영 문서와 실제 스키마 정합성 회복

현재 문제:

```txt
매뉴얼과 블로그 설명 문서 일부가 이미 폐기된 필드명과 예시를 유지하고 있다.
예: thumbnailPath, imagePath, htmlPath, pdfPath, DESIGN prefixTags 설명 등
```

목표:

```txt
운영 문서, 예시 콘텐츠, 실제 코드가 같은 규칙을 설명하도록 맞춘다.
신규 작성자가 오래된 예시를 복사해 잘못된 frontmatter를 만드는 일을 막는다.
```

실행 범위:

```txt
.agents/references/ol-home-content-management-manual.md
README.md
AGENTS.md 중 관련 설명
src/content/blog 내 구조 설명용 블로그 문서
docs 내 기획 문서 중 현재 운영 규칙을 설명하는 부분
```

세부 작업:

```txt
1. deprecated 필드 목록 확정
2. 현행 필드 명세표 작성
3. 매뉴얼/README/예시 문서 일괄 수정
4. "과거 규칙"은 삭제하거나 명시적으로 historical note 처리
```

완료 기준:

```txt
rg 검색 시 thumbnailPath, imagePath, htmlPath, pdfPath가
현재 운영 규칙으로 설명되는 문서에 남아 있지 않다.
```

예상 효과:

```txt
콘텐츠 작성 오류 감소
Codex 세션 간 일관성 향상
문서-코드 충돌로 인한 유지보수 비용 감소
```

### 3.2 이미지/자산 연결 규칙 통일

현재 문제:

```txt
TEXT는 상대경로 Astro asset 방식,
DESIGN은 assets -> public/generated 동기화 방식,
BLOG는 thumbnailAsset 또는 본문 첫 상대 이미지 탐색 방식이다.
```

목표:

```txt
컬렉션마다 자산 규칙이 완전히 달라서 생기는 인지 비용을 줄인다.
작성자는 "어디에 두고 어떤 필드명으로 연결하는가"를 한 가지 원칙으로 이해할 수 있어야 한다.
```

권장 방향:

```txt
1. 메타데이터 필드명은 thumbnailAsset / imageAsset / previewAssets / htmlAsset / pdfAsset로 고정
2. 저장 위치는 가능한 한 entry 폴더 내부 assets/ 또는 entry 로컬 파일 규칙으로 통일
3. 렌더링 쪽에서는 컬렉션별로 다른 입력을 공통 asset resolver가 해석
```

실행 범위:

```txt
src/content.config.ts
src/pages/blog/index.astro
src/pages/design/[slug].astro
src/pages/design/index.astro
scripts/sync-content-assets.mjs
자산 규칙 설명 문서 전체
```

세부 작업:

```txt
1. 컬렉션별 허용 자산 필드 정의표 작성
2. 공통 resolveAssetPath() 유틸 설계
3. BLOG 썸네일, DESIGN preview, TEXT cover를 같은 의미 체계로 재정렬
4. 자산 경로 표기에서 ./, assets/, plain filename 허용 범위 명확화
5. 필요한 경우 migration guide 작성
```

완료 기준:

```txt
모든 컬렉션이 공통 naming 규칙을 사용한다.
자산 표시 방식이 문서 한 장으로 설명 가능하다.
페이지 템플릿별 개별 예외 처리가 최소화된다.
```

예상 효과:

```txt
콘텐츠 등록 속도 향상
자산 누락/경로 오류 감소
렌더링 코드 단순화
```

### 3.3 자동 검증 스크립트 도입

현재 문제:

```txt
현재 품질 확인은 거의 npm run build에 의존한다.
빌드는 통과하지만 운영 규칙 위반은 그대로 통과할 수 있다.
```

목표:

```txt
frontmatter 일관성, 권리 정보 누락, 태그 형식 오류,
deprecated 필드 사용, 썸네일 누락 등을 자동으로 검출한다.
```

권장 명령 구조:

```txt
npm run lint:content
npm run check:frontmatter
npm run check:assets
npm run check
```

검사 항목 예시:

```txt
deprecated field 사용 여부
DESIGN tags prefix 형식
TEXT/STORY tags prefix 형식
thumbnailAsset/imageAsset 파일 실제 존재 여부
previewAssets 파일 실제 존재 여부
권리 정보 요구 조건
published 콘텐츠의 필수 메타 필드 존재 여부
```

실행 범위:

```txt
package.json
scripts/ 아래 신규 validator script
CI workflow 필요 시 확장
```

세부 작업:

```txt
1. content schema 외 운영 규칙 목록 정의
2. Node 기반 validator script 작성
3. 로컬 실행 명령 추가
4. 추후 GitHub Actions에 선택적 연결
```

완료 기준:

```txt
빌드 전 또는 별도 check 단계에서 운영 규칙 위반을 자동 감지할 수 있다.
새 콘텐츠 추가 후 수동 눈검사 의존도가 크게 줄어든다.
```

예상 효과:

```txt
사소한 운영 오류 조기 차단
콘텐츠 규모 증가 시 품질 유지 용이
```

```txt

태그 표현 규칙을 한 군데에서 수정할 수 있다.
```

예상 효과:

```txt
코드량 감소
```

### 3.6 저장소 위생 정리

현재 문제:

```txt
.gitignore에 .DS_Store가 있지만 이미 저장소 곳곳에 추적된 .DS_Store가 남아 있다.
운영 잡파일이 src, docs, public, dist 등 여러 경로에 퍼져 있다.
```

목표:

```txt
비소스 파일이 저장소에 남지 않도록 정리하고, 재유입을 방지한다.
```

실행 범위:

```txt
전체 저장소
.gitignore
필요 시 사전 검사 스크립트
```

세부 작업:

```txt
1. 추적 중인 .DS_Store 정리
2. 불필요한 로컬 산출물 추적 여부 재검토
3. pre-commit 또는 check 스크립트로 재유입 감시
4. docs/ 와 src/content/ 하위의 불필요한 잡파일 패턴 점검
```

완료 기준:

```txt
find . -name '.DS_Store' 기준 추적 대상이 남지 않는다.
이후 동일 파일 재유입을 자동 검출할 수 있다.
```

예상 효과:

```txt
저장소 신뢰도 향상
불필요한 diff 감소
협업 품질 향상
```

### 3.8 폰트/외부 리소스 로딩 전략 개선

현재 문제:

```txt
BaseLayout이 jsDelivr Pretendard와 Google Fonts를 함께 사용한다.
외부 의존이 많고, 초기 렌더링과 재현성 측면에서 다소 무겁다.
```

목표:

```txt
폰트 로딩 정책을 단순화하고,
장기적으로는 self-host 또는 공급원 최소화를 검토한다.
```

실행 범위:

```txt
src/layouts/BaseLayout.astro
public/fonts/ 사용 여부 검토
styles/tokens.css 및 global font stack
```

세부 작업:

```txt
1. 실제 사용 중인 폰트 세트 확인
2. 공급원 통합 또는 self-host 전환안 작성
3. preload/preconnect 정책 최소화
4. fallback stack과 CLS 영향 검토
```

완료 기준:

```txt
폰트 로딩 구성이 문서화되어 있고,
현재보다 단순한 공급 구조를 가진다.
```

예상 효과:

```txt
초기 로딩 안정성 향상

디자인 재현성 향상
```

## 4. 단계별 실행 계획

### Phase 1. 운영 규칙 수렴

기간 목표:

```txt
코드와 문서가 같은 말을 하게 만든다.
```

실행 항목:

```txt
3.1 운영 문서와 실제 스키마 정합성 회복
3.2 이미지/자산 연결 규칙 통일 설계
3.6 저장소 위생 정리
```

산출물:

```txt
최신 운영 매뉴얼
deprecated 필드 제거 목록
자산 규칙 표준안
정리된 저장소 상태
```

### Phase 2. 자동 검증과 구조 공통화

기간 목표:

```txt
사람이 기억하지 않아도 규칙 위반이 잡히는 구조를 만든다.
```

실행 항목:

```txt
3.3 자동 검증 스크립트 도입
3.4 태그/슬러그/계층 유틸 공통화
3.5 필터 UI와 클라이언트 스크립트 공통 모듈화
```

산출물:

```txt
check 스크립트
공통 유틸 레이어
재사용 가능한 필터 초기화 구조
```

### Phase 3. UI/스타일 품질 상향

기간 목표:

```txt
구조와 표현의 경계를 명확히 한다.
```

실행 항목:

```txt
3.7 스타일 레이어 정리
3.8 폰트/외부 리소스 로딩 전략 개선
```

산출물:

```txt
인라인 스타일 축소
더 명확한 CSS 역할 분리
단순화된 폰트 로딩 정책
```

## 5. 권장 작업 순서

실제 착수 순서는 다음을 권장한다.

```txt
1. 문서/예시 정리
2. deprecated 필드 제거 기준 확정
3. 자산 연결 표준안 확정
4. content validator 추가
5. 저장소 잡파일 정리
6. 공통 유틸 리팩터링
7. 필터 UI 공통화
8. 스타일/폰트 최적화
```

이 순서를 권장하는 이유는 다음과 같다.

```txt
문서와 규칙이 정리되지 않으면
자동 검증도 잘못된 기준을 강화할 수 있다.

검증이 먼저 잡혀야
그 뒤의 리팩터링이 구조 회귀를 만들지 않는다.

스타일과 폰트는 중요하지만
운영 규칙이 흔들리는 상태에서 마지막 우선순위가 맞다.
```

## 6. 성공 판단 기준

이 로드맵이 성공적으로 수행되면 프로젝트는 다음 상태에 가까워져야 한다.

```txt
새 콘텐츠를 등록할 때 참고해야 할 규칙이 한 벌로 정리되어 있다.
문서와 코드가 서로 충돌하지 않는다.
잘못된 frontmatter와 누락 자산을 자동 검출할 수 있다.
TEXT/STORY/DESIGN/BLOG가 공통 naming 규칙을 공유한다.
필터 UI와 태그 처리 로직이 재사용 가능한 수준으로 정리된다.
저장소에 운영 잡파일이 남지 않는다.
페이지 템플릿은 구조 중심, 스타일은 CSS 중심으로 분리된다.
```

## 7. 후속 작업 제안

이 문서 이후 실제 작업은 다음 순서로 이어가는 것이 적절하다.

```txt
1. P0 항목만 대상으로 한 첫 번째 실행 계획 문서 작성
2. deprecated field 제거와 운영 매뉴얼 정리부터 착수
3. content validator 초안 구현
4. 이후 공통 유틸과 필터 UI 리팩터링 진행
```
