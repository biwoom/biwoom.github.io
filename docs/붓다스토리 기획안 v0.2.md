문서 버전: v0.2
작성일: 2026-06-18
대상 프로젝트: OL HOME / OL STORY / OL ENTITY / OL NET / OL DESIGN / OL TEXT
기획 범위: 붓다스토리 공개 STORY 본문, STORY 인물카드, ENTITY 인물 상세 페이지, DESIGN 이미지 자산, TEXT 톺아보기, NET 검색·지도 탐색

## 0. v0.2 업데이트 핵심

v0.2는 현재 OL HOME 구조와 운영 판단을 반영하여 v0.1의 일부 전제를 조정한다.

```txt
1. 원전서사본은 OL HOME에 공개하지 않는다.
2. OL HOME에는 공개용 STORY 본문과 그 프론트메타를 반영한다.
3. Entity 원본 데이터에서 memoryPhrase 필드는 사용하지 않는다.
4. 공개 신설 메뉴명은 NET으로 통일한다.
5. ENTITY는 현재 개별 문서와 내부 데이터 구조를 유지한다.
6. NET은 ENTITY 문서를 검색·필터·지도와 함께 탐색하는 별도 메뉴다.
7. DESIGN은 기존 홈페이지 폴더 구조를 유지하고, 필요한 경우 스키마를 확장한다.
8. 붓다스토리의 파트명과 회차 제목은 `docs/toc/붓다스토리(Buddha Story) 목차 v0.1.md`를 우선 참고한다.
```

`docs/toc/붓다스토리(Buddha Story) 목차 v0.1.md`는 아직 완성된 최종 목차는 아니지만, 현재 붓다스토리의 큰 파트 구분과 원전 장절 대응을 정리한 기준 문서다. STORY 문서의 `part`, `partSlug`, `title`, `chapter`, `order`를 정할 때 이 문서를 먼저 확인하고, 실제 공개 원고 제목은 현재 작성된 대중서사본의 제목과 조율한다.

## 1. 기획의 핵심 방향

붓다스토리는 부처님 일대기를 단순히 다시 서술하는 전기 콘텐츠가 아니다.

붓다스토리는 독자가 부처님의 생애와 과거 보살행을 다음 흐름으로 따라가며 이해하고 기억하도록 설계된 웹 서사 콘텐츠다.

```txt
이야기 → 인물 → 톺아보기 → 관계망
```

각 단계의 역할은 다음과 같다.

```txt
STORY      = 이야기의 흐름
인물카드   = STORY 안에서 즉시 소비되는 인물 요약 인터페이스
ENTITY     = 인물·장소·개념 등 지식 객체의 원본 데이터와 개별 상세 페이지
NET        = ENTITY를 검색·필터·지도와 함께 탐색하는 공개 메뉴
TEXT       = 심화 해설과 톺아보기
DESIGN     = 삽화·캐릭터시트·스타일시트·이미지 자산 관리
```

이 기획안의 핵심은 다음 한 문장으로 정리할 수 있다.

> STORY는 인물을 장면 속에서 만나게 하고, 인물카드는 그 인물을 즉시 이해하게 하며, ENTITY는 그 인물의 원본 정보를 축적하고, NET은 그 원본 정보를 검색·지도와 연결해 탐색하게 하며, DESIGN은 인물과 장면을 시각화하고, TEXT는 그 의미를 깊게 해설한다.

## 2. 주요 역할 구분

### 2.1 STORY

STORY는 독자가 실제로 읽는 붓다스토리 본문이다.

담당 내용:

```txt
붓다스토리 회차별 서사 원고
여정 목록
본문 인물뱃지
STORY 인물카드 호출
DESIGN 삽화 호출
TEXT 톺아보기 링크
원전과 전승 안내
```

STORY 본문은 짧고 장면 중심이어야 한다.

```txt
짧은 단락
명확한 장면
감정의 흐름
인물의 내적 변화
교리의 자연스러운 삽입
과도한 학술 설명 배제
```

STORY는 논문도 강의록도 아니다. 독자가 부처님 생애의 사건을 따라가며 자연스럽게 질문을 일으키도록 하는 서사 공간이다.

---

### 2.2 STORY 인물카드

STORY 인물카드는 ENTITY 상세 페이지가 아니다.

STORY 인물카드는 STORY 안에서 바로 소비되는 “요약 인터페이스”다.

역할:

```txt
현재 장면의 등장인물 확인
인물의 짧은 소개와 장면상 역할 제공
이 장면에서의 역할 표시
ENTITY 상세 페이지로 이동
```

STORY 인물카드는 정보량이 적어야 한다.

기본 구성:

```txt
대표 이미지
이름
짧은 설명
이 장면에서의 역할
더 보기
```

예:

```txt
수메다

사라지지 않는 길을 찾기 시작한 사람.

명문 브라만 가문에서 태어나 지식과 재산을 모두 갖추었지만,
부모의 죽음과 선조들의 유산 앞에서 삶의 덧없음을 바라보기 시작한 인물.

이 장면에서:
재산은 남았지만 사람은 남지 않는다는 사실을 처음 깊이 응시한다.

더 보기 →
```

중요 원칙:

```txt
인물카드 = STORY 문맥에 맞춘 축약 뷰
ENTITY = 인물 원본 데이터와 상세 페이지
```

즉, 인물카드는 ENTITY 데이터를 STORY 문맥에 맞게 축약 렌더링한 화면이다.

---

### 2.3 ENTITY

ENTITY는 인물·장소·개념 등 지식 객체의 원본 데이터와 개별 상세 페이지다.

붓다스토리 v0.1에서는 우선 인물 중심으로 `entities/persons`를 구성한다.

담당 내용:

```txt
인물 명칭과 별칭
출신지와 거주지
가족관계
스승·제자관계
군신관계
전생인과관계
직업·재산·신분 등 간략 이력
외형적 특징
수행상태
등장 STORY 문서명과 문서경로
연결 TEXT 문서
연결 DESIGN 스타일시트
프로필 이미지
관련 삽화와 캐릭터시트
원전별 전승 차이
```

ENTITY는 단순 프로필이 아니라, 향후 NET 검색·관계망·스토리 연결·이미지 자산 연결의 기준점이다.

---

### 2.4 TEXT

TEXT는 「붓다스토리 톺아보기」를 담당한다.

담당 내용:

```txt
원전 비교
전승 차이
교리 해설
학술적 해설
현대적 의미 해석
부처님 생애의 사상적 의미
```

예:

```txt
수메다는 누구인가
보살도와 수기 사상
네 가지 장면과 고성제
출가와 가족의 문제
고행과 중도
성도와 사성제
초전법륜의 의미
```

STORY 본문에 모든 해설을 넣지 않는다.

STORY는 이야기로 남기고, 깊은 해설은 TEXT로 분리한다.

---

### 2.5 DESIGN

DESIGN은 붓다스토리의 모든 시각 자산을 관리하는 영역이다.

DESIGN은 단순한 이미지 저장소가 아니다. 삽화·캐릭터시트·스타일시트·인포그래픽을 축적하는 OL 프로젝트의 시각 라이브러리다.

담당 내용:

```txt
인물 프로필 이미지
캐릭터시트
장면 삽화
표지 이미지
복식 기준
색상 팔레트
시각 스타일시트
이미지 생성 프롬프트
금지 표현
수정 이력
이미지 자산 경로
```

원칙:

```txt
이미지는 STORY 폴더에 흩어 저장하지 않는다.
이미지는 DESIGN 콘텐츠로 일괄 관리한다.
STORY 본문은 DESIGN 이미지 경로를 호출하여 사용한다.
```

## 3. 전체 정보 구조

붓다스토리의 기본 정보 흐름은 다음과 같다.

```txt
STORY 문서
  ↓ primaryEntities / [[character:id]]
STORY 인물카드
  ↓ 요약 렌더링
ENTITY persons 상세 페이지
  ↓ 원본 인물 데이터
NET
  ↓ 검색·필터·지도 탐색
DESIGN 이미지 자산 / 스타일시트
  ↓ 프로필 이미지·삽화·캐릭터시트
TEXT 톺아보기
  ↓ 원전·전승·교리 해설
```

역할을 더 압축하면 다음과 같다.

```txt
STORY      = 만남
인물카드   = 즉시 이해
ENTITY     = 원본 데이터
NET        = 탐색
DESIGN     = 형상과 자산
TEXT       = 이해
```

## 4. 독자의 기본 독서 구조

독자는 다음 흐름으로 붓다스토리를 읽는다.

```txt
1. STORY 본문 읽기
2. 본문 속 인물뱃지 클릭
3. STORY 인물카드 확인
4. 더 보기 클릭
5. ENTITY 인물 상세 페이지 이동
6. NET에서 관련 STORY·TEXT·DESIGN 탐색
```

### 데스크탑

데스크탑에서는 3단 구조를 기본으로 한다.

```txt
좌측: 여정 목록
중앙: 이야기 본문
우측: 인물카드
```

본문에서 `[[character:sumedha]]` 같은 표식을 클릭하면 우측 인물카드가 해당 인물로 교체되거나 강조된다.

### 모바일

모바일에서는 우측 사이드바를 제거한다.

```txt
본문 속 인물뱃지 클릭
→ 인물카드 모달 출력
→ 더 보기 클릭
→ ENTITY 상세 페이지 이동
```

모바일에서는 큰 카드가 본문 흐름을 방해하지 않도록, 작은 인물뱃지와 모달 방식을 사용한다.

## 6. STORY 시리즈 index.md 프론트메터

경로:

```txt
src/content/story/buddha-story/index.md
```

예시:

```yaml
---
title: "붓다 스토리"
kind: series
subtitle: "부처님의 생애를 따라 읽는 이야기"
series: "붓다 스토리"
seriesSlug: "buddha-story"
seriesOrder: 10
category: "인물 · 행적"
version: "v0.1.0"
status: "draft"
description: "부처님의 생애와 과거 보살행을 이야기 형식으로 재구성한 OL STORY 시리즈입니다."
tags:
  - "인물/붓다"
  - "개념/보살도"
  - "형식/이야기불교"
tagAliases:
  "인물/붓다": ["Buddha", "부처님", "석가모니", "고타마 붓다"]
  "개념/보살도": ["Bodhisatta Path", "Bodhisattva Path", "보살의 길"]
  "형식/이야기불교": ["Buddhist Story", "불교서사"]
```

## 7. STORY 개별 문서 프론트메터

STORY 개별 문서는 현재 OL HOME 프론트메터 구조를 따른다.

예시:

```yaml
---
title: "수메다, 길을 묻기 시작하다"
kind: document

series: "붓다 스토리"
seriesSlug: "buddha-story"
seriesOrder: 10

part: "먼 과거의 서원"
partSlug: "part-1"
partOrder: 10

group: "수메다"
groupSlug: "sumedha"
groupOrder: 10

category: "붓다전기"

chapter: 1
order: 1

version: "v0.1.0"
status: "draft"
publishedAt: 2026-06-16

primaryEntities:
  - "sumedha"

description: "명문 브라만 가문에서 태어난 수메다가 부모의 죽음과 선조들의 유산을 마주하며 삶의 덧없음을 처음 깊이 바라보기 시작하는 장면입니다."

tags:
  - "인물/수메다"
  - "장소/아마라와띠"
  - "개념/보살도"
  - "개념/무상"
  - "개념/출가"
  - "경전/마하붓다왐사"
  - "형식/붓다스토리"

tagAliases:
  "인물/수메다": ["Sumedha", "수메다 브라만", "수메다 고행자"]
  "장소/아마라와띠": ["Amaravatī", "Amaravati", "아마라바티"]
  "개념/보살도": ["Bodhisatta Path", "Bodhisattva Path", "보살의 길"]
  "개념/무상": ["Anicca", "impermanence"]
  "개념/출가": ["Nekkhamma", "renunciation", "집을 떠남"]
  "경전/마하붓다왐사": ["Mahābuddhavaṃsa", "Maha Buddhavamsa", "Great Chronicle of Buddhas"]
  "형식/붓다스토리": ["Buddha Story", "OL STORY"]

license: "CC0"
rightsHolder: "비움"
lang: "ko"
published: true
---
```

현재 단계에서는 다음 확장 필드를 STORY 프론트메터에 넣지 않는다.

```yaml
layout
nav
topics
timeline
characters
deep_dive
sources
```

대신 다음 방식으로 처리한다.

```txt
layout       → STORY 리더 컴포넌트에서 처리
nav          → order 기반 자동 이전/다음 계산
topics       → tags로 통합
timeline     → 본문 또는 TEXT에서 설명
```

## 9. STORY 인물카드 렌더링 원칙

## 11. DESIGN 콘텐츠 구조

DESIGN은 붓다스토리의 모든 이미지 자산을 관리한다.

폴더 구조는 현재 OL HOME의 기존 DESIGN 운영 방식을 유지한다.

```txt
src/content/design/{slug}/index.md
src/content/design/{slug}/assets/
```

붓다스토리 관련 DESIGN도 우선 이 구조 안에 둔다. 인물 스타일시트, 캐릭터시트, 장면 삽화가 늘어나면 `type`, `primaryKind`, `format` 등 DESIGN 스키마를 확장하되, 중첩 라우트 구조로 급히 전환하지 않는다.

```txt

2. 캐릭터시트
3. 장면 삽화
```

## 13. 여러 원전 기반 제작 워크플로우

붓다스토리는 하나의 원전만 단순 번역하는 콘텐츠가 아니다.

하나의 이야기는 여러 원전과 전승에서 내용을 추출하고, 독자에게 자연스러운 하나의 이야기로 재구성할 수 있다.

이때 다음 제작 워크플로우를 따른다.

### 13.1 원전 수집

대상 장면과 관련된 원전을 수집한다.

예:

```txt
마하붓다왐사
붓다왐사
니다나까타
율장 전승
한역 불전
현대 연구서
```

### 13.2 사건 단위 추출

원전에서 사건 단위를 분리한다.

```txt
도시 배경
인물 출신
부모의 죽음
재산 상속
무상에 대한 사유
출가 결심
디빵까라 부처님과의 만남
수기
```

### 13.3 STORY 장면 구성

사건 단위를 독자 친화적 장면으로 재구성한다.

```txt
장면성
감정 흐름
인물 변화
불교적 의미
읽기 쉬운 문장
```

### 13.4 ENTITY 업데이트

STORY에 등장하거나 새롭게 의미가 강화된 인물은 필요에 따라 `entities/persons`를 업데이트한다.

업데이트 항목:

```txt
새 등장 STORY 경로 추가
인물 요약 수정
관계 정보 추가
수행상태 보완
원전별 전승 차이 기록
관련 DESIGN 경로 추가
관련 TEXT 톺아보기 추가
```

### 13.5 DESIGN 제작

STORY 장면과 ENTITY 인물 정보를 바탕으로 DESIGN 자료를 제작한다.

제작 대상:

```txt
인물 프로필 이미지
캐릭터시트
장면 삽화
표지 이미지
관계도 이미지
```

### 13.6 STORY 본문에 DESIGN 호출

완성된 DESIGN 이미지 자산을 STORY 본문에서 호출한다.

```md
[[design:scene:sumedha-treasure-house]]
[[design:person:sumedha-profile]]
```

### 13.7 TEXT 톺아보기 작성

STORY 본문에서 설명하지 않은 학술적·교리적 내용을 TEXT로 분리한다.

예:

```txt
수메다는 누구인가
수기란 무엇인가
보살도는 어떻게 시작되는가
마하붓다왐사의 전승 위치
```

`story/buddha-story/appendix/sources.md`에 해당 회차의 원전과 전승 정보를 추가한다.

## 14. NET 연계 기획

붓다스토리와 OL 전체가 확장되면, 상단 메인 메뉴에는 `NET`을 둔다.

NET은 ENTITY와 다른 메뉴다.

```txt
ENTITY = 개별 인물·장소·개념 문서와 원본 데이터
NET    = ENTITY 문서를 검색·필터·지도와 함께 탐색하는 공개 메뉴
```

NET이 담당하는 기능:

```txt
인물 검색
장소 검색
태그 필터
지도 기반 장소 탐색
관련 STORY·TEXT·DESIGN 표시
ENTITY 개별 문서로 이동
```

현재 `/entity/...` 개별 문서는 유지한다. 검색 결과나 NET 상세 패널에서 필요할 때 ENTITY 개별 문서로 진입할 수 있게 한다.

## 17. 인물 관계망 시각화 아이디어

인물 관계망은 NET의 장기 확장 기능이다.

v0.2 이전에는 관계망 그래프를 구현하지 않는다. 우선 ENTITY 문서에 관계 데이터를 축적하고, NET에서는 검색·필터·지도 탐색을 먼저 안정화한다.

향후 관계망은 단순 목록이 아니라, 인물들이 서로 어떻게 연결되어 있는지 보여주는 탐색 공간이어야 한다.

### 17.1 관계 버튼 방식

각 인물 상세 페이지 또는 NET 상세 패널에 관계 버튼을 둘 수 있다.

예:

```txt
가족
스승
제자
왕실
후원자
전생 인연
같은 STORY 등장
같은 원전 등장
```

수메다 페이지에서 `전생 인연` 버튼을 클릭하면 디빵까라 부처님 카드가 출력된다.

싯다르타 페이지에서 `가족` 버튼을 클릭하면 야소다라, 라훌라, 숫도다나, 마야, 마하빠자빠띠 고따미 카드가 출력된다.

### 17.2 관계 렌즈 방식

인물 관계망 그래프에 렌즈를 적용한다.

```txt
가족 렌즈
스승·제자 렌즈
왕실 렌즈
출가 공동체 렌즈
전생 인연 렌즈
대립과 시험 렌즈
후원 관계 렌즈
```

같은 인물이라도 렌즈에 따라 다른 관계가 강조된다.

예:

```txt
싯다르타 + 가족 렌즈
→ 야소다라, 라훌라, 숫도다나, 마야, 마하빠자빠띠

싯다르타 + 수행 렌즈
→ 알라라 깔라마, 웃다까 라마뿟따, 다섯 수행자

싯다르타 + 대립 렌즈
→ 마라
```

### 17.3 STORY 장면 기반 관계망

특정 STORY 문서를 기준으로 그 장면에 등장하는 인물만 관계망으로 보여준다.

예:

```txt
「수메다, 길을 묻기 시작하다」
- 수메다
- 부모
- 재무관
- 선조들
- 아마라와띠

「진흙길의 서원」
- 수메다
- 디빵까라 부처님
- 대중
```

이 방식은 독자가 “이 장면에서 누가 누구와 어떤 관계인가”를 직관적으로 이해하게 한다.

### 17.4 전생-현생 레이어

붓다스토리는 전생 인과관계가 중요하다.

따라서 관계망에 “전생-현생 레이어”를 둘 수 있다.

예:

```txt
수메다
→ 미래의 고타마 부처

디빵까라 부처님
→ 수메다에게 미래 성불을 수기한 부처

라훌라
→ 고타마 부처의 아들
→ 전생 인연이 있는 존재
```

이 기능은 자칫 복잡해질 수 있으므로 v0.1에서는 데이터 구조만 준비하고, 시각화는 후순위로 둔다.

### 17.5 원전별 관계 비교

같은 인물 관계가 원전마다 다르게 설명되는 경우, 관계망에서 원전별 보기 기능을 제공한다.

예:

```txt
니까야 기준 보기
마하붓다왐사 기준 보기
한역 전승 기준 보기
후대 불전문학 기준 보기
```

이 기능은 학술성이 강하므로 초기에는 TEXT 톺아보기에서 설명하고, 관계망 기능은 나중에 확장한다.

### 17.6 인물 카드 확장 패널

관계망의 노드를 클릭하면 작은 인물 카드가 열린다.

카드 구성:

```txt
이름
짧은 설명
관계 유형
관련 STORY
더 보기
```

예:

```txt
디빵까라 부처님

수메다의 미래를 비추어 준 과거불.

관계:
수메다에게 미래 성불을 수기함.

관련 STORY:
진흙길의 서원

더 보기 →
```

## 18. 관계 데이터 기본 구조

ENTITY 인물 문서의 관계 데이터는 다음 구조를 권장한다.

```yaml
relationships:
  family:
    - target: "rahula"
      relation: "아들"
      direction: "outgoing"
      certainty: "traditional"
      sources:
        - "Buddhist biographical tradition"

  teacherDisciple:
    - target: "ananda"
      relation: "제자"
      direction: "outgoing"

  political:
    - target: "bimbisara"
      relation: "왕과 수행자"
      direction: "mutual"

  karmic:
    - target: "dipankara-buddha"
      relation: "수기 인연"
      direction: "incoming"
      description: "디빵까라 부처님이 수메다에게 미래 성불을 수기함."

  storyCooccurrence:
    - target: "channa"
      story: "/story/buddha-story/part-1/03-renunciation/"
      relation: "출가 동행"
```

관계 데이터의 기본 원칙:

```txt
target = 연결 대상 인물 id
relation = 관계 이름
direction = 관계 방향
certainty = 전승 확실성
sources = 근거 원전 또는 전승
description = 독자용 설명
```

## 19. 제작 우선순위

v0.1에서는 모든 기능을 한 번에 구현하지 않는다.

현실적 제작 순서는 다음과 같다.

```txt
1. STORY 붓다스토리 index.md 작성
2. part-1 「먼 과거의 서원」 중 수메다 관련 3~6편 작성
3. STORY 문서 프론트메터 표준 확정
4. entities/persons 구조 확정
5. 핵심 인물 ENTITY 5개 작성
   - 수메다
   - 디빵까라 부처님
   - 싯다르타 고타마
   - 야소다라
   - 라훌라
6. DESIGN 인물 스타일시트 2개 작성
   - 수메다
   - 싯다르타 고타마
7. DESIGN 장면 삽화 2개 작성
   - 아마라와띠
   - 수메다의 보물창고
8. STORY 본문에서 DESIGN 이미지 호출 방식 테스트
9. STORY 인물카드 UI 설계
10. 모바일 인물카드 모달 설계
11. TEXT 톺아보기 2개 작성
   - 수메다는 누구인가
   - 보살도와 수기 사상
12. NET 진입 페이지 제작
13. NET 인물·장소 검색·필터 기능 추가
14. 기존 ENTITY 개별 문서 진입 연결 유지
15. 인물 관계망은 후속 버전에서 구현
```

## 20. v0.2의 핵심 원칙

붓다스토리 v0.2의 핵심 원칙은 다음과 같다.

```txt
1. STORY는 가볍게 유지한다.
2. STORY 인물카드는 요약 인터페이스로 둔다.
3. ENTITY는 인물 원본 데이터와 상세 페이지로 둔다.
4. entities/persons 구조를 기준으로 인물 데이터를 축적한다.
5. NET은 ENTITY 문서를 검색·필터·지도와 함께 탐색하는 별도 공개 메뉴로 둔다.
6. DESIGN은 기존 홈페이지 폴더 구조를 유지하되 필요한 스키마를 확장한다.
7. STORY는 DESIGN 이미지 경로를 호출하여 사용한다.
8. TEXT는 STORY의 학술적·교리적 해설을 담당한다.
9. 여러 원전에서 하나의 이야기를 구성할 경우, STORY·ENTITY·DESIGN·TEXT·NET 연결 정보를 함께 점검한다.
10. 인물 관계망은 NET의 장기 확장 기능으로 설계한다.
11. 현재 OL HOME 프론트메터와 충돌하지 않는 범위에서 점진적으로 확장한다.
```

최종적으로 붓다스토리는 단순한 부처님 일대기 콘텐츠를 넘어 다음 성격을 갖는다.

```txt
부처님의 생애를 읽는 STORY
인물을 기억하는 카드
인물 원본 데이터를 축적하는 ENTITY
인물·장소를 검색하고 지도와 연결하는 NET
시각 자산을 관리하는 DESIGN
의미를 깊게 해설하는 TEXT
인물의 인연을 확장 탐색하는 관계망
```

즉, 붓다스토리는 부처님 생애의 사건을 읽는 콘텐츠가 아니라, 부처님 생애 속 인물들을 만나고, 기억하고, 연결하고, 그 의미를 자기 삶으로 되돌아보는 OL의 대표 웹 서사 프로젝트가 된다.
